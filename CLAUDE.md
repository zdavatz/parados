# CLAUDE.md

Guidance for Claude Code when working in this repo.

## Project

Parados is a collection of standalone web board games by Walter Prossnitz, served as static files from `/var/www/game.ywesee.com/parados/` at `game.ywesee.com/parados`. Each game is one self-contained `.html` file (markup + `<style>` + `<script>`). No build, no package manager, no test framework, no dependencies. To run: open the file in a browser.

`index.html` is the landing page.

## Architecture

- **Single-file pattern** per game.
- **Localization by duplication:** language variants are separate files (e.g. `kangaroo.html` DE, `kangaroo_en.html`, `_jp`, `_cn`, `_ua`). When changing game logic, propagate to all variants.
- **Common JS shape:** `gameState` string state machine, 2D `board[y][x]`, `renderBoard()` + `initGame()`, `historyStack` + `undoMove()`, `localStorage` for prefs.

### `divided_loyalties.html` (+ `_en`) ‚Äî Divided Loyalties
Connect-4 with 6 colors. Two-player tile + bridge building. Non-square grid (`GRID_W` √ó `GRID_H`, computed per position with +2 margin for off-board placements). `_en` is a UI-only translation ‚Äî embedded CSV `position_name` values stay English (`Up the Stairs`, `Wind wheel`, ‚Ä¶) so `slugifyName` yields the same hash anchors in both files and shared links travel across language variants.

- Positions: external `divided_loyalties_starting_positions.csv` + embedded copy in HTML (app WebViews can't `fetch()` over `file://`, so the embedded copy is what ships). **Sync both when editing.** CSV format: `position_id,position_name,row,c0..cN`; cell value `'.'` = empty, `'o'` = placement dot, anything else = board token. Each unique token = one separately scored board.
- Game-length groups: built-ins are a contiguous 1‚Äì12 split into a **4-round set (ids 1‚Äì6)** and a **3-round set (ids 7‚Äì12)** ‚Äî Walter's design convention, *not* stored in the CSV. `showSetup()` renders full-width section headers ("4-Runden-Spiel" / "3-Runden-Spiel" / "Eigene Stellungen"; `_en`: "4-Round Game" / "3-Round Game" / "Custom Positions") driven by a `FOUR_ROUND_COUNT = 6` constant. New positions append to the 3-round set; if the boundary ever moves, change that constant in both HTML variants. The old standalone "Swiss Cross" (slug `swiss-cross`) was retired 2026-05-16, superseded by "SwissCross" (slug `swisscross`); numeric `#7` now resolves to Pyramid.
- Bridges: manual via double-click two endpoints (on-board placements). **Exception:** dot placements auto-form *every* justifying bridge (see "Multi-bridge auto-form" below). **Toggle selection model (Walter, 2026-05-27):** `handleStoneDblClick` now treats each dblclick as a TOGGLE on `bridgeSelection` (`includes(idx)` ‚Üí splice out; otherwise push). When `bridgeSelection.length === 2` the bridge auto-attempts. **Success** ‚Üí clear selection + draw. **Failure** ‚Üí `undo()` restores the snapshot (taken right after the second push, so both rings come back), then `flashStoneInvalid(idx)` red-flashes the last clicked stone. A 3rd click while two are ringed is refused via `flashStoneInvalid` (no state change). Walter's reasoning: the previous "second click fires the bridge attempt immediately" never let the second stone visibly ring, so silent rejections looked like the click was ignored ‚Äî "the system can't know which is the first stone, too many possible variants, so the player must keep control." `renderBridgePending` iterates `bridgeSelection` and adds/removes `.bridge-pending` (cyan outline) on every matching tile; switched from `cell.firstChild` to `cell.querySelector('.tile')` so the ring works on stones placed on dot cells (where future DOM children ‚Äî overlays, dot markers ‚Äî would otherwise break the lookup). **Amend rule (Walter, 2026-05-26):** after the system auto-draws the dot-placement bridge, the player may override it with a 2-stone toggle whose row *still contains the just-placed stone* ‚Äî the auto-bridge is removed and the player's chosen row takes its place. Tracked via `amendableBridgeId` (id of the auto bridge, or of the previous amend so further amendments still work). Reset on `endTurn`. The amend snapshot includes `amendableBridgeId`, so if the player's amend attempt is invalid (bad row, collision, etc.) `undo()` restores the auto-bridge. Walter's rationale: "the system picks the longest row by default, but a shorter 3-bridge in a different direction may score better via a region bonus ‚Äî the player must keep control."
- Off-board placement rule (Walter, 2026-05-26 ‚Äî corrected): a stone may be placed on a dot only if it completes a bridge in the same turn. There is **no** "max 1 off-board tile per row" limit ‚Äî multiple "dotted" stones may share a row, since each was already justified by completing a bridge of its own at placement time. `classifyRow` therefore checks no-adjacent-same-colour + colour-set membership *only*; do **not** re-add an `offBoard > 1` guard. The earlier ruleset that limited a row to one off-board stone was Walter's mis-reading of the rule.
- **No 2+ stone share with an existing bridge (Walter, 2026-05-27 ‚Äî generalised from the old "no 3‚Üí4 extension"):** "Br√ºcken k√∂nnen nicht sp√§ter verl√§ngert werden!" ‚Äî `tryFormBridge`, `findBridgeContaining`, and `findAutoBridges.isValidCandidate` now run the same `sharesTwoPlus` check: any new bridge sharing **2 or more** stones with an active (non-cut) bridge is rejected. This subsumes (a) re-forming the same row, (b) extending a 3 to a 4 (existing 3 ‚äÇ new 4), and (c) overlapping 3-bridges along the same line ‚Äî e.g. existing `[A,B,C]` + dropping `D` adjacent to `C` used to silently form a second `[B,C,D]` (visually a 4-bridge). Single-stone overlap is still allowed and is how perpendicular crosses + the 5-stone multi-bridge case (upper `[A,B,C]` + lower `[C,D,E]` share only `C`) work. The `b.cut` filter is preserved: a cut bridge frees its line for re-bridging (geometric crossings are still blocked by `checkCollision`).
- **Multi-bridge auto-form (Walter, 2026-05-27):** when a stone is placed on a dot, `findAutoBridges(idx, player)` enumerates every justifying row through `idx` and `handleClick` runs each through `tryFormBridge` ‚Äî a single placement can complete *multiple* bridges (e.g. dropping the middle blue into a 5-stone vertical column forms both the upper 3 and the lower 3, which share only the placed stone). Algorithm: per direction, take the longest valid "before" row (idx at row.last) + longest valid "after" row (idx at row.first); if neither exists, fall back to ONE "middle" row (idx in the interior) per direction. **Placement-reject rule:** if every candidate row turns out unbuildable (typically because each would cross an opponent scoring bridge), `handleClick` calls `undo()` and returns ‚Äî the off-board rule requires an actual bridge in the same turn, and a placement that justified a row only in theory is illegal. `amendableBridgeId` now tracks the LAST auto-built bridge (the amend single-bridge override still works on it).
- **Nobody crosses an existing bridge (Walter, 2026-05-27):** red bridges are pushed into `activeBridges` with `player:'red'` (previously they only existed as SVG lines), so `checkCollision` can see them. Both branches of `tryFormBridge` now call `checkCollision(c, false)`. **Red branch:** on collision ‚Äî `cut` ‚Üí return; `player==='red'` ‚Üí return ("Rot kann Rot nicht schneiden ‚Äî sonst w√§ren keine Verteidigungs¬≠strukturen m√∂glich"); else `destroyBridge` (cut behaviour). **Scoring branch:** any collision ‚Üí return (Walter: "the yellow or blue bridges cannot cross over the red ones ‚Äî nobody can"). The defensive value of red comes from cutting at **construction time** only; once a red bridge is on the board, scoring lines must route around it. The `skipRed` parameter is retained on `checkCollision` but is now always `false` from both call sites (left in place in case a future rule needs the asymmetric behaviour). `recomputeScores` filters out `b.player === 'red'` so the red entries don't count toward `bridgeScore`/`regionScore`.
- Cut bridges stay in `activeBridges` with `cut:true` ‚Äî score deducted, but still block new bridges (`checkCollision` walks them).
- Score split: `bridgeScore` (3=1pt, 4=2pts) + `regionScore` (first scoring bridge per player√óboard whose *inner* cells touch a region = +1).
- Undo + save: `undoStack` snapshots before each user-input mutation; `localStorage["dl_savegame"]` auto-saves; `‚ñ∂ Letztes Spiel fortsetzen` in setup; manual JSON export/import + WhatsApp share.
- Editor: custom positions with independent W√óH, brush bar of board tokens + Punkt + eraser, persisted to `localStorage["dl_custom_positions"]`. CSV export + WhatsApp share use shared `buildPositionCSV()`.
- Stones nearly fill the cell (`--tile-size: calc(var(--cs) - 4px)`). Bridge endpoints offset by `0.50¬∑cellSize` (don't graze stones). Bridges over stones (`#bridge-layer z-index:20`, `.tile z-index:15`).

**Don't touch:**
- **No `wa.me` redirect** in any share-fallback path (`shareCSVViaWhatsApp`, `shareGameJSONViaWhatsApp`, `shareOnWhatsApp`). Android WebView can't dispatch `whatsapp://` ‚Üí `ERR_UNKNOWN_URL_SCHEME`. Fallback = download + alert.
- **No `text:` field** on `navigator.share` calls that pass `files:` ‚Äî Android WhatsApp drops the file when both are present.
- **Don't "rebalance" `BLUE_BATCH` / `YELLOW_BATCH`** ‚Äî Walter confirmed the per-batch asymmetry (yellow gets 1 purple vs blue's 3) is intentional role-symmetry.
- **Color brightness hierarchy:** blue/yellow may be bright; red/violet/orange stay darker (red = "evil" defensive color); green darkest. Bridges always one Material step darker than their stone.
- **Board grays:** `S‚Üí#3a3a3a` (darkest), `W‚Üí#6a6a6a` (middle), `G‚Üí#9a9a9a` (lightest) ‚Äî not monotonic by cell count, per Walter's order.
- **Don't reintroduce auto-dots** in the editor ‚Äî Walter places them explicitly with the Punkt brush.

### `makalaina.html` / `makalaina_remote.html` ‚Äî MAKA LAINA
2-player disc collection on 12√ó12. 54 discs (36 primary numbered 1‚Äì6 in R/B/Y, 18 secondary in G/O/P). Draft system: after each collection, draw 1 from bag and place in your LOS. Sacrifice/relinquish mechanic when stuck (relinquish capped at 2/player). Winner = most colors won (tiebreak: highest single-color score). Remote uses PeerJS with full state sync.

- **Opening flow (Walter's rule ‚Äî fair under any disc layout):** 24 random discs are dropped onto the empty board first. Then **Player 1 places both collectors** (one click ‚Üí Black, second ‚Üí White). Once both are down, **Player 2 picks which colour they want** ‚Äî Player 1 plays the other and goes first. State machine: `SETUP ‚Üí PLACEMENT ‚Üí P2_CHOOSE ‚Üí PLAY`. In the remote variant, host = P1 (the only one allowed to click during PLACEMENT); the joiner = P2 picks via `p2PicksColor()` which sends a `p2_pick_color` PeerJS message + a follow-up `state_sync`. **No curated starting positions** ‚Äî Walter removed them; opening fairness comes from P2's colour pick, not from balanced disc layouts.
- **Undo:** `‚Ü∂ Undo move` button in `#game-actions` (local only ‚Äî `makalaina_remote.html` has no undo). Every `saveSnapshot(desc)` captures full state (`cells`, `bag` contents, `gameState`, `scores`, `playerFlags`, `collectors`, `collectedDiscs`, `relinquished`/`relinquishCount`, `draftChosen`, `pendingExtraTurn`, `selectedIdx`, `sacrificeRange`, `sacrificingPlayer`, `isFirstMove`). `undoMove()` pops the last snapshot and restores live state from the new head, exits review mode. Disabled when `moveHistory.length <= 1` or `gameState === 'END'`. `applyPositionData()` and `startRandom()` both call `resetGameState()` so test-playing from the editor after a prior game starts fresh.
- **Snapshot timing (Walter bugs, 2026-05-29 ‚Äî fixed in both variants):** `processCollection` now snapshots at the *resolved* terminal state of the turn, not mid-way. (a) **Draft branch:** snapshot is taken *after* `gameState='DRAFT_PLACE'` + `draftChosen=bag.pop()`, so undoing a draft *placement* lands back in DRAFT_PLACE with the disc in hand (`undoMove` re-shows the draft panel when the restored state is `DRAFT_PLACE`), ready to re-place ‚Äî previously it dropped to PLAY with no disc. (b) **No-draft (bag-empty) branch + `relinquishTurn`/`performRelinquish`:** snapshot is taken *after* the `currentPlayer` switch, so the saved state records whose turn it now is. Snapshotting *before* the switch made undo restore the previous mover as current player ‚Äî the "it's the other player's turn after undo" bug, which only surfaced once the bag was empty (with a draft, the post-placement snapshot masked it).
- **Free escape move (Walter's rule, 2026-05-29):** a trapped player (`hasValidMove`==false) with **no primary discs to sacrifice** no longer forfeits. `freeEscapeMove(player)` grants a free collector move of up to **2** squares (sets `SACRIFICE_MOVE` + `sacrificeRange=2`, no disc lost, no opponent pick), reusing the existing `SACRIFICE_MOVE`/`getReachableCells`/`handleSacrificeMove` plumbing. Falls back to relinquish/end only if even a 2-square move is impossible (truly boxed in). Applied at **both** trigger points in each file: the manual `sacrificeTurn`/`doSacrifice` no-discs branch *and* the auto-forced-end inside `updateUI`. Rationale (Walter): being trapped early used to be a near-certain loss; a guaranteed escape removes the incentive to trap the opponent in the opening. The "2" is a tunable constant ‚Äî change it in `freeEscapeMove` in both variants.
- **Custom-position editor (local only):** `mlOpenEditor(null)` opens the 12√ó12 painter. Brush tokens ‚Äî `R1..R6`/`B1..B6`/`Y1..Y6` for primary discs, `G`/`O`/`P` for secondaries; paint keyed by `y*12+x`. Customs are shown numbered ("‚òÖ Position 1", "‚òÖ Position 2", ‚Ä¶) in setup-order; the H1 shows just "MAKA LAINA" (no `‚Äî name` suffix) when a custom is played. ### `capovolto.html` ‚Äî Capovolto

## Share on WhatsApp button

Every game has a Share-on-WhatsApp button right of the `<h1>` (green `#25D366`, `gap:14px`).

`shareOnWhatsApp()` flow:
1. If `window.location.protocol === "file:"`, rewrite to `https://game.ywesee.com/parados/<filename>` (WebView container paths are useless to recipients). **Keep this guard for any new game.**
2. Try `navigator.share({ text: msg })` first.
3. If `navigator.share` is missing, copy `msg` to the clipboard and `alert()` the user to paste it into WhatsApp. **NEVER `wa.me` (Walter, 2026-05-31).**

**No `wa.me/?text=` anywhere ‚Äî text shares too (Walter, 2026-05-31):** the old fallback redirected to `whatsapp://send/?text=‚Ä¶` and Android WebView (and some in-app browsers) can't dispatch the scheme ‚Üí `net::ERR_UNKNOWN_URL_SCHEME` ("Web page not available"). The replacement is the same pattern as file shares: try `navigator.clipboard.writeText(msg)`, fall back to a hidden `<textarea>` + `document.execCommand('copy')`, then `alert()` a localized "Message copied ‚Äî paste it into WhatsApp" (DE/EN/JP/CN/UA per file `lang=`). If even the textarea copy fails, `prompt(note, msg)` shows the text so the user can long-press to copy. Applied to all 17 game files (`democracy√ó3`, `kangaroo√ó5`, `divided_loyalties√ó2`, `makalaina√ó2`, `capovolto`, `frankenstein`, `rainbow_blackjack√ó3`). The `navigator.share` short-circuit stays first so iOS/Mac/desktop browsers keep the native share sheet ‚Äî only the no-share fallback changed.

For file shares (CSV / saved game JSON), use `navigator.share({ files:[file], title:'‚Ä¶' })` ‚Äî no `text:`, no `wa.me` fallback. See divided_loyalties guardrails.

## Hash deep-links per position

`divided_loyalties.html` (+ `_en`) and `kangaroo.html` (+ 4 language variants) accept a URL hash that auto-launches a specific starting position on load and on `hashchange`. The pattern: `findPositionByHash(hash)` resolves either a slug or a numeric index, and `setPositionTitle(label)` updates an inline `<span id="position-title">` in the H1 plus `document.title`. `shareOnWhatsApp` appends the active position's slug to `shareUrl` and includes the position name in the message body.

**Cross-language slugs:** kangaroo levels use English mnemonic slugs (`first-steps`, `the-block`, `shifting`, `staircase`, `scattered`, `slant`, `the-cross`) stored in `LEVEL_DATABASE[*].id` ‚Äî identical across all 5 language variants, so `kangaroo_jp.html#the-cross` and `kangaroo_en.html#the-cross` resolve to the same level. ## Server-side
