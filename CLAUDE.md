# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Parados is a collection of standalone web-based board games by Walter Prossnitz, hosted at `game.ywesee.com/parados`. Each game is a single self-contained HTML file with embedded CSS and JavaScript — no build system, no dependencies, no framework.

## Development

There is no build step, no package manager, and no test framework. To run a game, open its `.html` file in a browser. The project is served as static files from `/var/www/game.ywesee.com/parados/`.

## Architecture

**Single-file pattern:** Each game is one `.html` file containing all markup, styles (in `<style>`), and logic (in `<script>`). There are no shared libraries or external dependencies.

**Landing page:** `index.html` is the main entry point listing all games with play buttons (links open in new tabs).

**Localization by duplication:** Language variants are separate files (e.g., `kangaroo.html` for German, `kangaroo_en.html` for English, `kangaroo_jp.html` for Japanese, `kangaroo_cn.html` for Chinese, `kangaroo_ua.html` for Ukrainian). When modifying game logic, changes must be propagated to all language variants.

**Games:**
- `kangaroo.html` (+ `_en`, `_jp`, `_cn`, `_ua`) — DUK (The Impatient Kangaroo): 21st century successor to Rushhour. Grid-based puzzle where Kangy the kangaroo collects food (green dishes) and water (blue dishes) for her family by jumping over them. Symmetric "catapult" jumps with color alternation. Before every jump, the player may move one dish by one square. **Dish move — double-click shortcut (Walter playtester feedback, 2026-05-07):** the original flow ("erst Knopf 🖐️ Schale verschieben → Schale wählen → Zielfeld") was reported as tedious — "immer auf Schale verschieben mühsam". After that, a single click on an adjacent empty cell completes the move via the existing `handleDishMove` path. The 🖐️ button remains as a fallback (visible affordance + works on platforms where dblclick is unreliable). `.cell` got `touch-action: manipulation` so double-tap on mobile fires `dblclick` instead of triggering browser zoom. Same change applied verbatim to all 5 language variants; rules text updated in each (DE/EN/JP/CN/UA) under "Wichtig!" / "Important!" / "重要提示！" / "Важливо!" with a 💡 Tipp/Tip/ヒント/小贴士/Підказка paragraph. Kangy rotates to face available jump directions; with multiple options she wobbles between them (800ms cycle, 3 full cycles then stops, smooth CSS transitions). Supports LEVEL, RANDOM, EDITOR, and REPLAY modes. RANDOM ("Take a Chance") has three placement-zone levels: Center 6×6 (12 dishes, pad=2 on 10×10), Inner 8×8 (20 dishes, pad=1), Full board (34/48 dishes, no pad) — Kangy always has access to the entire board. Color-balance validation (`isColorBalanced`) rejects placements where either color clusters on one side — each board half (left/right, top/bottom) must have ≥30% of each color's total; reshuffles up to 100 times. As a fallback for any imbalance the player still notices in practice, the rules modal also tells them to click "Neues Spiel" if a board half has 3 or more excess dishes of one color (added per Walter, 2026-05-07; tip paragraph in all 5 language variants right after the double-click tip). Has a `LEVEL_DATABASE` array for predefined puzzles across three difficulty tiers: Anfänger (3 levels, 2–3 dishes), Mittelstufe (3 levels, 6–8 dishes), and Fortgeschritten (1 level, 12 dishes). Anfänger and Fortgeschritten levels have two starting positions (`startHint` / `startHintB`) selectable via A/B toggle buttons; Mittelstufe levels have only one starting position each (A/B toggle auto-hidden). Mittelstufe order: Treppe (easiest, Kangy center-bottom), Verstreut (middle, Kangy top-left), Schräglage (hardest, Kangy bottom-right). For the Kreuz (cross) advanced level, Start A = center (easy), Start B = corner (hard). REPLAY mode: loads exported CSV game logs via file input, reconstructs the initial board state from move data (two-pass algorithm: pass 1 discovers initial dish positions from MOVE_DISH and JUMP actions, pass 2 generates board snapshots), and provides playback controls (first/prev/play-pause/next/last) with adjustable speed (200ms–3s per move). Visual highlights mark source (yellow) and target (cyan) positions for each action. Supports multi-game CSV files with a game selector dropdown.
- `divided_loyalties.html` — Divided Loyalties: connect 4 with 6 colours. Two-player tile placement and bridge-building. Tiles can be loyal in one direction and disloyal in another. Responsive layout: viewport meta tag, dynamic cell sizing via JS (`calcCellSize()`), bridges redrawn on resize (`redrawBridges()`). UI panel stacks below grid on narrow screens. **Non-square grid per position**: the play grid tracks `GRID_W` (column count / row stride) and `GRID_H` (row count) **independently**, computed in `startWithPosition()` as `GRID_W = maxX + 2; GRID_H = maxY + 2;` from the chosen position's cell coordinates. The +2 on each axis gives a 1-cell margin for stones placed just outside a board (an "excellent rule" per Andres Kuusk: tiles may be placed one cell off-board if they immediately form part of a bridge). Walter's Oktopus is 15×9 — the grid renders 16×10, no wasted square padding. Outer-edge detection in `init()` uses `±GRID_W` for N/S neighbours and `GRID_H - 1` for the south boundary; `±1` for W/E and `GRID_W - 1` for the east boundary. But `checkCollision` still walks `activeBridges`, which means **anything else built across the cut bridge is blocked** — both new scoring bridges and new red bridges. - `democracy.html` / `democracy_remote.html` — Democracy in Space: area majority game based on the Electoral College concept. Local version is pass-and-play for 2 players; remote version adds PeerJS networking for online play. Responsive layout: viewport meta tag, dynamic cell sizing via JS (`calcCellSize()`), sidebar stacks below grid on narrow screens. Uses HTML canvas for connection lines.
- `makalaina.html` / `makalaina_remote.html` — MAKA LAINA: 2-player disc collection strategy on a 12x12 grid. Responsive layout: viewport meta tag, dynamic cell sizing via JS (`calcCellSize()`), UI panel stacks below grid on narrow screens. 54 discs total (36 primary in red/blue/yellow with numbers 1-6, 18 secondary in green/orange/purple). 6 hidden, 24 start on board, 24 drafted in. Draft system: after each collection, 1 disc drawn from bag; player must place it on the board in their line of sight (LOS). Drafted discs in LOS cannot be collected immediately (collection requires discs outside LOS). LOS highlighting is off by default, toggled via "Show Line of Sight" button — when on, highlights LOS cells during both regular play and draft placement. For the first 24 turns disc count stays at 24 on board (each collected disc replaced by 1 from bag); after bag empties, count decreases. Sacrifice mechanic (only available when stuck with no valid moves): opponent chooses which of your collected primary discs to remove (score reduced), then you move collector freely up to that disc's value in squares (Chebyshev distance). Relinquish is limited to 2 times per player per game (button shows remaining count); sacrifice only appears when stuck. When stuck with no relinquishes left, sacrifice is forced. If stuck with no relinquishes and no primary discs to sacrifice, the turn counts as a forced relinquish toward game-end detection. Number 1 = extra turn, Number 6 = future discs of that color worthless (shown in color-matched bordered box). Color-aware scoring: each color name in the scoring table is displayed in its actual color; worthless boxes use the disc's own color for border/background. CSV export via `downloadCSV()` available at any time during play or after game end — exports full move history with per-move scores, flags, collector positions, and disc counts. Move history replay: all moves are recorded as snapshots; at game end, history navigation controls (first/prev/next/last) appear allowing players to step through the entire game move by move. Game continues until all discs collected. Plan from the get go, stay flexible. After board setup, players choose who starts (Black or White) via a "Who starts?" panel — no automatic first-player advantage. First-move draft restriction: the first draft disc cannot be placed in both players' LOS simultaneously. Winner determined by color-majority: whoever wins the most colors wins the game (not total points). Tiebreaker: highest score in any single color. Winner announced via overlay with color wins count, and the status bar at game end shows a persistent short summary via `computeWinner()` (e.g. "Black wins! 4–2", "Black wins (tiebreak)! 3–3", or "Tie 3–3") so players don't need to count colors themselves; remote version shows the same with a P1/P2 player-indicator dot. Scoring table shows leader indicators (black/white/gray squares) per color. Curated starting positions: `makalaina_starting_positions.csv` contains 10 playtested positions (collector coords + 24 disc placements) transcribed from physical board photos; format is vertical CSV with one row per element (position_id, element, x, y, color, number). Both local and remote versions have a setup panel at game start where players choose "Random" or a numbered position (1–10). Selecting a curated position places both collectors and all 24 discs from the data, builds the draft bag from remaining discs (minus 6 hidden), and starts play immediately. In the remote version, the host selects the position and syncs state to the joiner. Remote version (`makalaina_remote.html`) has full feature parity with local version: uses PeerJS for networking, with message types for clicks, sacrifice request/pick (`sacrifice_request`/`sacrifice_pick`), relinquish, and full state sync (including bag and collected discs).
- `capovolto.html` — Capovolto: Othello on steroids! 2-player disc-flipping strategy game on a polyomino grid. Players place 18 polyominoes, then alternate placing numbered discs across 6 rounds. Core mechanic: flipping 4-disc lines (2 of each color) to change values. Opponent's last disc value determines your direction constraint (even=H/V, odd=diagonal). Scoring by value majority per polyomino.
- `rainbow_blackjack.html` (+ `_en`, `_remote`) — Rainbow Blackjack: "Colorful 21!" 2-player tower-building strategy game. 36 colored stones (1-6 in red/blue/yellow/green/orange/purple) plus 6 gray jokers (3 per player). Player 1 gets primary colors, Player 2 secondary. Program removes 3 random stones per player. Remaining 18 auto-arranged by program into two 3×3 grids (9 each). Grid stones hidden from opponent — players memorize positions from the look phase. Each round: both players pick a row in any direction (first row freely choosable, then one of the other two; direction locked per grid) and announce colors OR numbers, then alternate placing 1 stone at a time (3 turns per round) into 6 shared color towers aiming for 21. KEY MECHANIC: Each colored stone can be placed in its own tower or the two adjacent towers on the color wheel (Red→Purple/Orange, Orange→Red/Yellow, Yellow→Orange/Green, Green→Yellow/Blue, Blue→Green/Purple, Purple→Blue/Red). Gray jokers copy previous stone's value, can't be first or consecutive. Gray replacement mechanic: if joker is on top and adding your stone would exceed 21, you may remove the joker and place your stone (if it gets closer to 21). Round starters: P1,P2,P1,P1,P2,P1. SCORING: Each tower shows owner dot (red=P1, green=P2). First to own 3 towers at exactly 21 wins instantly; otherwise most towers wins. Local version uses pass-and-play; remote version (`_remote`) uses PeerJS for separate screens.
- `frankenstein.html` — Frankenstein: "Where's that green elbow?" A "frankly memorable" memory game for 1-4 players, ages 7+. Players flip face-down cards to find body parts and build a complete Frankenstein body from the middle outward. 32/48/64 cards for 2/3/4 players. Solo mode uses 2-player variant where one person controls both Frankies. No left/right distinction — any arm/leg/hand/foot card fits either side. All 4 torso parts are identical and interchangeable. Strict torso adjacency: each limb requires its specific adjacent torso part (right arm→torso_ul, left arm→torso_ur, left leg→torso_bl, right leg→torso_br); head needs any top torso. Body part colors: green, orange, purple, turquoise; torso is grey. Game continues until ALL Frankies are complete (first to finish wins). Terminology: "misses" = card you can't use (2 per turn), "mistakes" = miss on a previously-seen card (optional tracking). Features build-order constraints, toggleable mistake tracking, joker cards, and CSS-drawn SVG body part art. Color-specific head designs with creepy Frankenstein personality: green=wild spiky hair + crazed bloodshot eyes with veins, stitched scar over left eye, teeth-baring grimace, orange=dense overlapping curly circles + asymmetric mad eyes (one big, one squinting), temple screw, unhinged wide sinister grin, purple=tall mohawk with shaved-side stubble + stitched scar across entire face, glassy mad eye, fangs, forked snake tongue, blue=angular flat-top with lightning scar + sunken eyes with dark rings underneath, jagged teeth, extra cheek stitches, neck bolts. All head outlines use consistent face-width at x=20/x=80 so any upper fits any lower. Body part shapes designed for maximum distinguishability: upper legs have anatomical thigh shape tapering from wide hip to narrower knee with kneecap arc at bottom, lower legs have asymmetric calf muscle bulge tapering dramatically to thin ankle, lower arms are thin straight parallel lines, hands have 5 spread fingers fanning from wrist (no palm curve), feet are wide flat L-shapes with toe bumps. **Build modes:** Classic (default, torso→outward), Reverse (extremities→inward), or Mixed (alternate directions per player). Classic pre-places 3 torso parts (grey); Reverse pre-places 1 hand + 1 foot + 1 upper_head (colored) instead — all 4 torso parts go in the deck. Same total card count (15/player) in all modes. Rules modal documents reverse build chains. Victory screen shows [R] tag for reverse players. Game-over screen shows "View Frankies" button to dismiss overlay and admire completed bodies before starting a new game.

**Common code patterns across games:**
- Global state variables with a `gameState` string for state machine flow
- 2D array `board[y][x]` for grid representation
- `renderBoard()` for DOM updates, `initGame()` for setup
- `historyStack` array with `undoMove()` for undo support
- `localStorage` for persisting user preferences (e.g., rules acceptance)
- CSV export via `downloadCSV()` for gameplay analytics logging — POSTs to `cgi-bin/save_csv.py` (saves to `csv/` directory as `gamename_HHMM_dd.mm.yyyy.csv`) and also triggers a local browser download
- Modal-based rules display with localStorage-gated auto-show; every rules modal has a close button at the bottom ("Verstanden!" / "Got it!" / "Schließen")
- Rules button placed at the very top of the page (above the title) for immediate visibility
- **Share on WhatsApp** button placed to the **right of the game title heading** (flex row with `gap:14px`, WhatsApp green `#25D366`). Calls `shareOnWhatsApp()` which builds a message with the game name, a `shareUrl` (see below), and the iOS/Android app links (`https://apps.apple.com/us/app/parados/id6760842713` and `https://play.google.com/store/apps/details?id=com.ywesee.parados`), then opens `https://wa.me/?text=<encoded>` in a new tab via `window.open(url, '_blank')` — `wa.me` auto-falls back to web.whatsapp.com if the WhatsApp app isn't installed. **`shareUrl` rewrites file:// to https://** — when the page is loaded inside the iOS/Android WebView, `window.location.href` is the local container path (`file:///var/mobile/Containers/Data/.../games/<name>.html` on iOS, equivalent on Android), which is useless to recipients. The first line of every `shareOnWhatsApp()` checks `window.location.protocol === "file:"` and substitutes `https://game.ywesee.com/parados/<filename>` so the link always works. Keep this guard when adding new games. Present in all games; rainbow_blackjack variants have three button instances (setup/game/scoring screens) since each has its own `<h1>`

**Server-side CGI:** `cgi-bin/save_csv.py` is a Python 3 CGI script that accepts POST requests with `{game, csv}` JSON and saves CSV files to the `csv/` directory. Filename format: `gamename_HHMM_dd.mm.yyyy.csv` with collision-avoidance suffixes. The `csv/` directory is owned by `www-data`. Apache CGI is enabled via `cgid` module with `ExecCGI` on the `cgi-bin/` directory.
