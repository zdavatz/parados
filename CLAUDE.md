# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Parados is a collection of standalone web-based board games by Walter Prossnitz, hosted at `game.ywesee.com/parados`. Each game is a single self-contained HTML file with embedded CSS and JavaScript — no build system, no dependencies, no framework.

## Development

There is no build step, no package manager, and no test framework. To run a game, open its `.html` file in a browser. The project is served as static files from `/var/www/game.ywesee.com/parados/`.

## Architecture

**Single-file pattern:** Each game is one `.html` file containing all markup, styles (in `<style>`), and logic (in `<script>`). There are no shared libraries or external dependencies.

**Landing page:** `index.html` is the main entry point listing all games with play buttons (links open in new tabs).

**Localization by duplication:** Language variants are separate files (e.g., `kangaroo.html` for German, `kangaroo_en.html` for English, `kangaroo_jp.html` for Japanese, `kangaroo_cn.html` for Chinese, `kangaroo_ua.html` for Ukrainian). When modifying game logic, changes must be propagated to all language variants.

**Games:**
- `kangaroo.html` (+ `_en`, `_jp`, `_cn`, `_ua`) — DUK (The Impatient Kangaroo): 21st century successor to Rushhour. Grid-based puzzle where Kangy the kangaroo collects food (green dishes) and water (blue dishes) for her family by jumping over them. Symmetric "catapult" jumps with color alternation. Before every jump, the player may move one dish by one square. Kangy rotates to face available jump directions; with multiple options she wobbles between them (800ms cycle, 3 full cycles then stops, smooth CSS transitions). Supports LEVEL, RANDOM, EDITOR, and REPLAY modes. RANDOM ("Take a Chance") has three placement-zone levels: Center 6×6 (12 dishes, pad=2 on 10×10), Inner 8×8 (20 dishes, pad=1), Full board (34/48 dishes, no pad) — Kangy always has access to the entire board. Color-balance validation (`isColorBalanced`) rejects placements where either color clusters on one side — each board half (left/right, top/bottom) must have ≥30% of each color's total; reshuffles up to 100 times. Has a `LEVEL_DATABASE` array for predefined puzzles across three difficulty tiers: Anfänger (3 levels, 2–3 dishes), Mittelstufe (3 levels, 6–8 dishes), and Fortgeschritten (1 level, 12 dishes). Anfänger and Fortgeschritten levels have two starting positions (`startHint` / `startHintB`) selectable via A/B toggle buttons; Mittelstufe levels have only one starting position each (A/B toggle auto-hidden). Mittelstufe order: Treppe (easiest, Kangy center-bottom), Verstreut (middle, Kangy top-left), Schräglage (hardest, Kangy bottom-right). For the Kreuz (cross) advanced level, Start A = center (easy), Start B = corner (hard). REPLAY mode: loads exported CSV game logs via file input, reconstructs the initial board state from move data (two-pass algorithm: pass 1 discovers initial dish positions from MOVE_DISH and JUMP actions, pass 2 generates board snapshots), and provides playback controls (first/prev/play-pause/next/last) with adjustable speed (200ms–3s per move). Visual highlights mark source (yellow) and target (cyan) positions for each action. Supports multi-game CSV files with a game selector dropdown.
- `divided_loyalties.html` — Divided Loyalties: connect 4 with 6 colours. Two-player tile placement and bridge-building. Tiles can be loyal in one direction and disloyal in another. Responsive layout: viewport meta tag, dynamic cell sizing via JS (`calcCellSize()`), bridges redrawn on resize (`redrawBridges()`). UI panel stacks below grid on narrow screens. **Non-square grid per position**: the play grid tracks `GRID_W` (column count / row stride) and `GRID_H` (row count) **independently**, computed in `startWithPosition()` as `GRID_W = maxX + 2; GRID_H = maxY + 2;` from the chosen position's cell coordinates. The +2 on each axis gives a 1-cell margin for stones placed just outside a board (an "excellent rule" per Andres Kuusk: tiles may be placed one cell off-board if they immediately form part of a bridge). Walter's Oktopus is 15×9 — the grid renders 16×10, no wasted square padding. All cell-index math uses `GRID_W` as the row stride: `idx = y * GRID_W + x`, `x = idx % GRID_W`, `y = Math.floor(idx / GRID_W)`. Outer-edge detection in `init()` uses `±GRID_W` for N/S neighbours and `GRID_H - 1` for the south boundary; `±1` for W/E and `GRID_W - 1` for the east boundary. The meta-grid's CSS uses `gridTemplateColumns = \`repeat(${GRID_W}, var(--cs))\`` and `gridTemplateRows = \`repeat(${GRID_H}, var(--cs))\``. **Off-board placement dots — manual** (Walter's request): the play grid no longer auto-renders a dot on every empty cell. Instead, the position designer places dots explicitly in the editor (Punkt brush) and they ride along on the position via a `dots: [[x,y]…]` field. CSS: `#meta-grid` has a `background: #fff` (the play area is plain white once a Startstellung is loaded — Walter: "the rest of the area is unnecessary so can just be left white"); `.off-board.has-dot::before` renders the 5px dark dot — an `.off-board` cell without `has-dot` is completely blank (no border, no fill). `init()` adds the `has-dot` class only to off-board cells whose flat index is in `playDots` (a `Set` populated by `startWithPosition()` from `pos.dots`). Built-in/CSV positions without a `dots` entry simply have no dots — Walter intends to add them per position via the editor. The rule is unchanged and still documented in the rules modal under section 4 (Außenfelder & Boards): stones may be placed on dot cells only if doing so completes a bridge in the same turn. **Starting positions via embedded CSV + external file**: `divided_loyalties_starting_positions.csv` is fetched at startup in `loadPositions()`; on failure it falls back to `HARDCODED_POSITIONS`, which is no longer a hand-coded array but the result of `parsePositionsCSV(EMBEDDED_POSITIONS_CSV)` — a copy of the full CSV pasted into the HTML as a template literal. **Why both:** iOS/Android/Rust app WebViews load the HTML via `file://` and `fetch('…csv')` either fails (no CORS/file access) or has no CSV file next to the HTML, so without an embedded copy app users only see hardcoded positions. The external CSV file stays in the repo so Walter can edit it and see changes live in the browser; when CSV content changes, **sync the embedded copy in the HTML too** (the embedded text is what ships in app bundles via the next "Spiele aktualisieren" / GitHub-update of the HTML file). **5 positions** (Walter's editor-designed Whatsapp drops, 2026-05-05): 1 Stairs, 2 Octopus, 3 Wind wheel, 4 Gentle kiss, 5 Utopia. The earlier 10 positions (Windrad, Raute, Asymmetrisch, Zwei-Säulen, H, Kreuz, Grid 1–4) were removed at Walter's request. All 5 use the editor's manual-dot convention (`o` cells in CSV); grids range from 15 to 17 columns wide. The CSV header is c0..c16. `parsePositionsCSV()` reads the grid-visualization CSV format: `position_id, position_name, row, c0, c1, c2, …, c16` (each cell its own column — lets Walter count left-to-right in a spreadsheet; c16 fits Stairs which is 17 columns wide). Row width is per-row — the parser uses `parts.slice(cellStart)`, so individual rows can be shorter than the header without padding. `parsePositionsCSV` is a function declaration (hoisted), so `HARDCODED_POSITIONS = parsePositionsCSV(EMBEDDED_POSITIONS_CSV)` runs synchronously near the top of the script before `STARTING_POSITIONS` is initialized. `'.'` = empty; any other value is a board id (convention: `S1,S2,…` = black 3×3, `W1,W2,…` = white 3+2+1 pyramid, `G1,G2,…` = grey 4×3). **Each unique token = one separate board** (no flood-fill merging) — so two 3×3s drawn as `S1` and `S2` next to each other stay two separately-scored regions with a visible seam, even when touching; critical for scoring per region. Boards are passed around as `{token, cells:[[x,y]…]}` objects; `startWith()` converts to flat indices, tracks `boardTokens[]` parallel to `boards[]`. **Size-class board colors** (Walter's aesthetic — all gray, no near-black/near-white, so the stones carry all the color): size class derived from token's first letter (S→large, G→medium, W→small), with cell-count fallback (9=large, 7–12=medium, ≤6=small). Walter's lightness order is **3×3 darkest → 3×2 (W) middle → 3×4 (G) lightest** — note this is NOT monotonic by cell count (small W is darker than larger G). Hex: large `#3a3a3a`, small `#6a6a6a`, medium `#9a9a9a`. Outer frames `rgba(255,255,255,0.25–0.35)` on dark fills, `rgba(0,0,0,0.30)` on light fills (subtle, not contrasting white-on-black). Inner borders very faint (`rgba(255,255,255,0.08–0.10)` on dark / `rgba(0,0,0,0.12)` on light). Outer edges detected per-cell in `init()` via neighbor lookup in a `cellBoard` map; emitted as a `data-outer` attribute (`N S W E`) matched by CSS `[data-outer~="..."]` selectors. Background is muted forest green `#3d5245` (distinct from the green tile). **Stone palette:** `--blue` `#2196F3`, `--yellow` `#FFEB3B`, `--red` `#F44336` (unchanged — blue/yellow are player identities, red is the "evil" color), `--green` `#2E7D32` (darker), `--purple` `#BA68C8` (lighter), `--orange` `#FFA726` (lighter). **Walter's brightness hierarchy** (do not violate when tweaking colors): blue and yellow may be relatively bright; red, violet, and orange should stay on the darker side (red because it is the "evil" defensive color, violet/orange for visual balance with the muted green board); green stays the darkest. Walter has explicitly confirmed the current violet and orange shades — do not lighten or darken them further. **Bridges must always be one Material step darker than their stone** (currently blue 700 vs 500, yellow 600 vs 500, red 700 vs 500) so they read as "darker but same hue"; if you ever bump a stone's brightness, bump the matching bridge in the same direction to preserve the gap. **Piece counts per position** (verified with Walter photo-by-photo): Windrad 5S+4W=69 cells, Raute 5S+4W=69, Asymmetrisch 3S+2G+4W=75, Zwei-Säulen 2S+2G+4W=66, H 5S+4W=69, Kreuz 5S+4W=69. Positions are Walter's intent transcribed from his sources (6 photos + grids_combined.xlsx); exact distances and shape adjustments are his to set via direct CSV edit. **Stone size:** `--tile-size: calc(var(--cs) - 4px)` — stones nearly fill the cell (~91% on 44px) per Walter's "fast die ganze Fläche" preference; do not shrink back to `cs - 6px`. **Bridge geometry:** bridge endpoints offset by `cellSize * 0.50` from each end-stone center (in both `redrawBridges()` and `processBridge()`) so the line ends at the cell edge (~2px clear of the stone) instead of grazing it. Walter's note: bridges should ideally not touch the outer stones at all, and never more than 1/4 of a stone, otherwise long bridges visually appear to cut other bridges. **Bridge stroke colors** (slightly darker than stones for legibility, per Walter): blue `#1976D2`, yellow `#FDD835` (Material 600 — explicitly NOT Yellow 700 which drifts toward orange), red `#D32F2F`. **Inventory tap target:** the inv-tile click handler is attached to the surrounding `.inv-slot` (parent), not to the small 28×28 `.inv-tile` itself — the slot includes the count number too, so the whole vertical slot accepts taps. Walter reported on mobile that tapping a color "did nothing" because the 28px tile was too easy to miss; do not move the handler back to the inner tile. **Inventory color labels + swatch fix:** each slot renders `[colored swatch] [count] [German color name]` vertically. Color names come from `COLOR_NAMES_DE = { blue:'Blau', yellow:'Gelb', purple:'Lila', orange:'Orange', green:'Grün', red:'Rot' }`. The `.inv-tile.<color>` background rules were missing for years (only `.tile.<color>` existed) — without them the swatches rendered as colorless dark rectangles and players couldn't tell which count belonged to which color. Both the missing CSS and the explicit name labels were Walter's fix request: "no one except me knows what these numbers refer to." **Drei Sätze (batch rule):** instead of all 36 stones at once, each player gets 12 stones at a time across 3 batches. Per-batch composition: blue gets `{blue:4, purple:3, green:2, red:2, orange:1}` (`BLUE_BATCH`), yellow gets `{yellow:4, orange:3, green:2, red:2, purple:1}` (`YELLOW_BATCH`) — exactly 1/3 of the old totals. `blueBatch`/`yellowBatch` track 1..3; `maybeRefillBatch()` runs in `endTurn()` after the player switch and refills any inventory that just hit 0 (until batch === `TOTAL_BATCHES`). Use `{...BLUE_BATCH}` (spread) when initializing/refilling so the constant isn't mutated. **Position editor (Walter's request — set up own start positions):** the setup overlay shows built-ins + custom positions; an "✏" button on every tile opens `openEditor(pos)` (built-ins are *cloned* with `customId: null`, customs are edited in place via `editingCustomId`); a "**➕ Neue Stellung erstellen**" button opens a blank editor. Walter's Oktopus is 15×9 — different W and H is fully supported end-to-end (editor → save → play). Cells painted with the • Punkt brush show a centered black dot (~22% of cell size). - `democracy.html` / `democracy_remote.html` — Democracy in Space: area majority game based on the Electoral College concept. Local version is pass-and-play for 2 players; remote version adds PeerJS networking for online play. Responsive layout: viewport meta tag, dynamic cell sizing via JS (`calcCellSize()`), sidebar stacks below grid on narrow screens. Uses HTML canvas for connection lines.
- `makalaina.html` / `makalaina_remote.html` — MAKA LAINA: 2-player disc collection strategy on a 12x12 grid. Responsive layout: viewport meta tag, dynamic cell sizing via JS (`calcCellSize()`), UI panel stacks below grid on narrow screens. 54 discs total (36 primary in red/blue/yellow with numbers 1-6, 18 secondary in green/orange/purple). 6 hidden, 24 start on board, 24 drafted in. Draft system: after each collection, 1 disc drawn from bag; player must place it on the board in their line of sight (LOS). Drafted discs in LOS cannot be collected immediately (collection requires discs outside LOS). LOS highlighting is off by default, toggled via "Show Line of Sight" button — when on, highlights LOS cells during both regular play and draft placement. For the first 24 turns disc count stays at 24 on board (each collected disc replaced by 1 from bag); after bag empties, count decreases. Sacrifice mechanic (only available when stuck with no valid moves): opponent chooses which of your collected primary discs to remove (score reduced), then you move collector freely up to that disc's value in squares (Chebyshev distance). Relinquish is limited to 2 times per player per game (button shows remaining count); sacrifice only appears when stuck. When stuck with no relinquishes left, sacrifice is forced. If stuck with no relinquishes and no primary discs to sacrifice, the turn counts as a forced relinquish toward game-end detection. Number 1 = extra turn, Number 6 = future discs of that color worthless (shown in color-matched bordered box). Color-aware scoring: each color name in the scoring table is displayed in its actual color; worthless boxes use the disc's own color for border/background. CSV export via `downloadCSV()` available at any time during play or after game end — exports full move history with per-move scores, flags, collector positions, and disc counts. Move history replay: all moves are recorded as snapshots; at game end, history navigation controls (first/prev/next/last) appear allowing players to step through the entire game move by move. Game continues until all discs collected. Plan from the get go, stay flexible. After board setup, players choose who starts (Black or White) via a "Who starts?" panel — no automatic first-player advantage. First-move draft restriction: the first draft disc cannot be placed in both players' LOS simultaneously. Winner determined by color-majority: whoever wins the most colors wins the game (not total points). Tiebreaker: highest score in any single color. Winner announced via overlay with color wins count, and the status bar at game end shows a persistent short summary via `computeWinner()` (e.g. "Black wins! 4–2", "Black wins (tiebreak)! 3–3", or "Tie 3–3") so players don't need to count colors themselves; remote version shows the same with a P1/P2 player-indicator dot. Scoring table shows leader indicators (black/white/gray squares) per color. Curated starting positions: `makalaina_starting_positions.csv` contains 10 playtested positions (collector coords + 24 disc placements) transcribed from physical board photos; format is vertical CSV with one row per element (position_id, element, x, y, color, number). Both local and remote versions have a setup panel at game start where players choose "Random" or a numbered position (1–10). Selecting a curated position places both collectors and all 24 discs from the data, builds the draft bag from remaining discs (minus 6 hidden), and starts play immediately. In the remote version, the host selects the position and syncs state to the joiner. Remote version (`makalaina_remote.html`) has full feature parity with local version: uses PeerJS for networking, with message types for clicks, sacrifice request/pick (`sacrifice_request`/`sacrifice_pick`), relinquish, and full state sync (including bag and collected discs).
- `capovolto.html` — Capovolto: Othello on steroids! 2-player disc-flipping strategy game on a polyomino grid. Players place 18 polyominoes, then alternate placing numbered discs across 6 rounds. Core mechanic: flipping 4-disc lines (2 of each color) to change values. Opponent's last disc value determines your direction constraint (even=H/V, odd=diagonal). Scoring by value majority per polyomino.
- `rainbow_blackjack.html` (+ `_en`, `_remote`) — Rainbow Blackjack: "Colorful 21!" 2-player tower-building strategy game. 36 colored stones (1-6 in red/blue/yellow/green/orange/purple) plus 6 gray jokers (3 per player). Player 1 gets primary colors, Player 2 secondary. Program removes 3 random stones per player. Remaining 18 auto-arranged by program into two 3×3 grids (9 each). Grid stones hidden from opponent — players memorize positions from the look phase. Each round: both players pick a row in any direction (first row freely choosable, then one of the other two; direction locked per grid) and announce colors OR numbers, then alternate placing 1 stone at a time (3 turns per round) into 6 shared color towers aiming for 21. KEY MECHANIC: Each colored stone can be placed in its own tower or the two adjacent towers on the color wheel (Red→Purple/Orange, Orange→Red/Yellow, Yellow→Orange/Green, Green→Yellow/Blue, Blue→Green/Purple, Purple→Blue/Red). Gray jokers copy previous stone's value, can't be first or consecutive. Gray replacement mechanic: if joker is on top and adding your stone would exceed 21, you may remove the joker and place your stone (if it gets closer to 21). Round starters: P1,P2,P1,P1,P2,P1. SCORING: Each tower shows owner dot (red=P1, green=P2). First to own 3 towers at exactly 21 wins instantly; otherwise most towers wins. Local version uses pass-and-play; remote version (`_remote`) uses PeerJS for separate screens.
- `frankenstein.html` — Frankenstein: "Where's that green elbow?" A "frankly memorable" memory game for 1-4 players, ages 7+. Players flip face-down cards to find body parts and build a complete Frankenstein body from the middle outward. 32/48/64 cards for 2/3/4 players. Solo mode uses 2-player variant where one person controls both Frankies. No left/right distinction — any arm/leg/hand/foot card fits either side. All 4 torso parts are identical and interchangeable. Strict torso adjacency: each limb requires its specific adjacent torso part (right arm→torso_ul, left arm→torso_ur, left leg→torso_bl, right leg→torso_br); head needs any top torso. Body part colors: green, orange, purple, turquoise; torso is grey. Game continues until ALL Frankies are complete (first to finish wins). Terminology: "misses" = card you can't use (2 per turn), "mistakes" = miss on a previously-seen card (optional tracking). Features build-order constraints, toggleable mistake tracking, joker cards, and CSS-drawn SVG body part art. Color-specific head designs with creepy Frankenstein personality: green=wild spiky hair + crazed bloodshot eyes with veins, stitched scar over left eye, teeth-baring grimace, orange=dense overlapping curly circles + asymmetric mad eyes (one big, one squinting), temple screw, unhinged wide sinister grin, purple=tall mohawk with shaved-side stubble + stitched scar across entire face, glassy mad eye, fangs, forked snake tongue, blue=angular flat-top with lightning scar + sunken eyes with dark rings underneath, jagged teeth, extra cheek stitches, neck bolts. All faces have eyebrows and jaw outlines. Ears split between upper/lower halves for funny mix-and-match combos (pointy/round/pierced/bolt). All head outlines use consistent face-width at x=20/x=80 so any upper fits any lower. **Server-side CGI:** `cgi-bin/save_csv.py` is a Python 3 CGI script that accepts POST requests with `{game, csv}` JSON and saves CSV files to the `csv/` directory. Filename format: `gamename_HHMM_dd.mm.yyyy.csv` with collision-avoidance suffixes. The `csv/` directory is owned by `www-data`. Apache CGI is enabled via `cgid` module with `ExecCGI` on the `cgi-bin/` directory.
