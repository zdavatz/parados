# CLAUDE.md

Guidance for Claude Code when working in this repo.

## Project

Parados is a collection of standalone web board games by Walter Prossnitz, served as static files from `/var/www/game.ywesee.com/parados/` at `game.ywesee.com/parados`. Each game is one self-contained `.html` file (markup + `<style>` + `<script>`). No build, no package manager, no test framework, no dependencies. To run: open the file in a browser.

`index.html` is the landing page.

## Architecture

- **Single-file pattern** per game.
- **Localization by duplication:** language variants are separate files (e.g. `kangaroo.html` DE, `kangaroo_en.html`, `_jp`, `_cn`, `_ua`). When changing game logic, propagate to all variants.
- **Common JS shape:** `gameState` string state machine, 2D `board[y][x]`, `renderBoard()` + `initGame()`, `historyStack` + `undoMove()`, `localStorage` for prefs.

## Games

### `kangaroo.html` (+ `_en`, `_jp`, `_cn`, `_ua`) — DUK (The Impatient Kangaroo)
Grid puzzle. Kangy jumps over green/blue dishes ("catapult" jumps, color alternation). Before each jump, the player may move one dish by one square. Modes: LEVEL, RANDOM, EDITOR, REPLAY.

- Dish move: 🖐️ button OR double-click a dish then click target (mobile-friendly shortcut).
- RANDOM has 3 zone sizes (6×6 / 8×8 / full); `isColorBalanced` enforces ≥30% of each color per half, with player escape hatch ("Neues Spiel" if a half has ≥3 excess of one color).
- `LEVEL_DATABASE` has 3 tiers (Anfänger / Mittelstufe / Fortgeschritten); A/B start toggles where defined.
- REPLAY loads exported CSV game logs (two-pass reconstruction).

**Don't touch:** dish-move shortcut wiring (`ondblclick` + `touch-action: manipulation`) — Walter playtested.

### `divided_loyalties.html` (+ `_en`, `_jp`, `_cn`, `_ua`) — Divided Loyalties
Connect-4 with 6 colors. Two-player tile + bridge building. Non-square grid (`GRID_W` × `GRID_H`, computed per position with +2 margin for off-board placements). `_en`/`_jp`/`_cn`/`_ua` are **UI-only translations** (DE original → EN → JP/CN/UA added 2026-06-30 to reach Walter's Ukrainian + Asian contacts; built as byte-identical copies of `_en` with only display strings localized via parallel translator agents, then validated for CSV/slug byte-identity, JS syntax, and guardrails). **Critical invariant for ALL variants:** embedded CSV `position_name` values stay English (`Up the Stairs`, `Wind wheel`, …) so `slugifyName` yields the same hash anchors in every file — shared deep-links (`#staircase`, `#rules`, …) travel across all 5 language variants. JS logic + code comments are byte-identical across variants; only `<html lang>`, `<title>`, visible text, `COLOR_NAMES_DE` *values*, the turn/set labels, section headers, alerts, bridge-refuse reasons, and the share message preamble are translated (URLs, `${...}` interpolation, board codes S1/G4/W, `[A,B,C]` notation, and names Andres Kuusk/Walter are kept). The JP/CN/UA variants localize the "Share on WhatsApp" button text (kangaroo keeps it English — a deliberate divergence, revertible if Walter wants parity).

- Positions: external `divided_loyalties_starting_positions.csv` + embedded copy in HTML (app WebViews can't `fetch()` over `file://`, so the embedded copy is what ships). **Sync both when editing.** CSV format: `position_id,position_name,row,c0..cN`; cell value `'.'` = empty, `'o'` = placement dot, anything else = board token. Each unique token = one separately scored board.
- Game-length groups: built-ins are a contiguous 1–12 split into a **4-round set (ids 1–6)** and a **3-round set (ids 7–12)** — Walter's design convention, *not* stored in the CSV. `showSetup()` renders full-width section headers ("4-Runden-Spiel" / "3-Runden-Spiel" / "Eigene Stellungen"; `_en`: "4-Round Game" / "3-Round Game" / "Custom Positions") driven by a module-level `FOUR_ROUND_COUNT = 6` constant. New positions append to the 3-round set; if the boundary ever moves, change that constant in both HTML variants. The old standalone "Swiss Cross" (slug `swiss-cross`) was retired 2026-05-16, superseded by "SwissCross" (slug `swisscross`); numeric `#7` now resolves to Pyramid.
- **The group actually drives round count (Walter, 2026-06-01 — was a silent no-op before):** `TOTAL_BATCHES` (the number of Sätze, each = 12 stones) is **4** for the 4-round set, **3** for the 3-round set + customs. It used to be a hardcoded `const TOTAL_BATCHES = 3`, so the group split was *purely cosmetic* and every position stopped after 3 Sätze — Wind wheel (id 3, a 4-round position) ended at "Satz 3/3" instead of 4/4. Now `TOTAL_BATCHES` is a `let`, set per game in `startWithPosition()` via `roundsForPosition(pos)`. **`roundsForPosition` matches by `pos.name`** (and returns 3 for anything with a `customId`), *not* by object identity — resume/import pass a deserialized **copy** of the position, so `STARTING_POSITIONS.indexOf` would miss; `findIndex(p => p.name === pos.name)` survives the JSON round-trip (DL position names are English in **both** variants, so this matches identically in DE + EN). `TOTAL_BATCHES` is **not** saved into the savegame — it's recomputed on load, so old 3-capped saves gracefully extend to 4. `maybeRefillBatch`, the "Satz X/Y" turn indicator, and the rules-modal text all read it.
- Bridges: **always** manual via double-click two endpoints — on-board *and* off-board (dot) placements alike (Walter, 2026-06-02; see "Manual dot bridges" below). Dot placements used to auto-form every justifying bridge; that whole auto-form path is gone. **Toggle selection model (Walter, 2026-05-27):** `handleStoneDblClick` now treats each dblclick as a TOGGLE on `bridgeSelection` (`includes(idx)` → splice out; otherwise push). When `bridgeSelection.length === 2` the bridge auto-attempts. **Success** → clear selection + draw. **Failure** → `undo()` restores the snapshot (taken right after the second push, so both rings come back), then `flashStoneInvalid(idx)` red-flashes the last clicked stone. A 3rd click while two are ringed is refused via `flashStoneInvalid` (no state change). Walter's reasoning: the previous "second click fires the bridge attempt immediately" never let the second stone visibly ring, so silent rejections looked like the click was ignored — "the system can't know which is the first stone, too many possible variants, so the player must keep control." `renderBridgePending` iterates `bridgeSelection` and adds/removes `.bridge-pending` (cyan outline) on every matching tile; switched from `cell.firstChild` to `cell.querySelector('.tile')` so the ring works on stones placed on dot cells (where future DOM children — overlays, dot markers — would otherwise break the lookup). **Amend rule (Walter, 2026-05-26 — SUPERSEDED 2026-06-02, now inert):** this overrode the auto-drawn dot bridge with a 2-stone toggle. With auto-form removed there is nothing to amend — `amendableBridgeId` is now never set to a non-null value, so the `if (amendableBridgeId !== null)` block in `handleStoneDblClick` is dead code (left in place, harmless, in case auto-form ever returns). The player simply forms the bridge they want directly. The old rationale ("the system picks the longest row by default, but a shorter 3-bridge may score better — the player must keep control") is now satisfied structurally: the player picks *every* bridge.
- Off-board placement rule (Walter, 2026-05-26 — corrected): a stone may be placed on a dot only if it completes a bridge in the same turn. There is **no** "max 1 off-board tile per row" limit — multiple "dotted" stones may share a row, since each was already justified by completing a bridge of its own at placement time. `classifyRow` therefore checks no-adjacent-same-colour + colour-set membership *only*; do **not** re-add an `offBoard > 1` guard. The earlier ruleset that limited a row to one off-board stone was Walter's mis-reading of the rule.
- **No 2+ stone share with an existing bridge (Walter, 2026-05-27 — generalised from the old "no 3→4 extension"):** "Brücken können nicht später verlängert werden!" — `tryFormBridge`, `findBridgeContaining`, and `findAutoBridges.isValidCandidate` now run the same `sharesTwoPlus` check: any new bridge sharing **2 or more** stones with an active (non-cut) bridge is rejected. This subsumes (a) re-forming the same row, (b) extending a 3 to a 4 (existing 3 ⊂ new 4), and (c) overlapping 3-bridges along the same line — e.g. existing `[A,B,C]` + dropping `D` adjacent to `C` used to silently form a second `[B,C,D]` (visually a 4-bridge). Single-stone overlap is allowed only as a **junction** (the 5-stone multi-bridge case: upper `[A,B,C]` + lower `[C,D,E]` share only `C`, which is an endpoint of each). **Superseded note:** `sharesTwoPlus` still skips `b.cut`, which used to mean "a cut bridge frees its line for re-bridging." That is **no longer true** as of the 2026-06-02 grid-exact rewrite — `bridgesCrossedBy` iterates **all** bridges incl. cut ones, so a collinear re-bridge over a cut line shares edge midpoints → blocked, consistent with Walter's "scoring cannot cover ANY bridge." The `b.cut` skip in `sharesTwoPlus` is now effectively dead (the geometry gate decides); left in place, harmless. (If Walter ever wants cut lines re-bridgeable again, `bridgesCrossedBy` would need to skip `b.cut` for collinear overlaps — but NOT for crossings, since cut bridges still block those.)
- **Manual dot bridges — NO auto-form (Walter, 2026-06-02; reverses "Multi-bridge auto-form"):** placing a stone on a dot now *only places the stone* — `handleClick`'s off-board branch builds **nothing**. The player forms the justifying bridge(s) by double-click, exactly like on-board bridges, through the single tested `handleStoneDblClick`→`tryFormBridge` path. **Why the reversal:** auto-form had to *guess* which (and how many) bridges the player meant. It enumerated every justifying row through `idx` and built each, preferring the longest (4 over 3). That silently created bridges the player never wanted — most damagingly a 4 that crosses an existing bridge at a shared stone when the player wanted the shorter, non-crossing 3 ("it insisted on the 4, I couldn't change it"), or an extra downward 3 alongside the wanted upward 4. Walter: *"only 1 bridge has to be placed, but the programme can't figure out which — so the player must keep control."* The other games (🦘, MakaLaina, Elbow) never guess and are bug-free; DL was the only one auto-building, hence the only one producing "senseless" bridges. **Off-board rule enforcement** moved to `endTurn`: a dot stone with **no non-cut bridge running through it** (`!activeBridges.some(b => !b.cut && b.row.includes(lastPlaced))`) is removed + the tile refunded — was `!bridgePlaced`, which a manual flow could wrongly satisfy with an *unrelated* bridge. The **placement gate** is unchanged: `wouldFormBridge(idx)` (→ `findAutoBridges` + `bridgeWouldForm`) still runs so a dot only lights up / accepts a tile when at least one *legal* bridge is possible there — but it no longer *builds* anything, it only validates that the player will have a move. `findAutoBridges`/`bridgeWouldForm` therefore survive (gate + highlight only); the auto-build loop, its `formedAny`/`undo()` reject, and `amendableBridgeId` assignment in `handleClick` are gone.
- **Highlighted dot must accept the tile — highlight ↔ placement consistency (Walter, 2026-06-01):** *(Context note: as of 2026-06-02 the dot no longer auto-builds — see "Manual dot bridges". The `findAutoBridges`/`bridgeWouldForm` machinery described here is retained for the **highlight + placement gate** only; "accept the tile" now means "at least one legal bridge is possible here," after which the player forms it manually.)* a turquoise (`valid-placement`) dot that *refused* a tile was two bugs. **(1) Longest-only shadowing:** `findAutoBridges` kept only the **longest** before/after row per end; if that 4-row crossed an existing bridge (so `tryFormBridge` rejected it on collision), the buildable shorter 3-row was never offered → `formedAny=false` → `undo()`. Fix: each end now pushes **both** the 4- and the 3-row (longest first); `tryFormBridge`'s `sharesTwoPlus` guard stops the 4 from *also* spawning its contained 3 when the 4 builds, so "longest wins" is preserved while the 3 is a real fallback. **(2) Collision-blind highlight:** `wouldFormBridge` (the highlight *and* the placement gate) used `findBridgeContaining`, which never ran `checkCollision` — so a dot whose only line crossed a bridge was highlighted yet correctly refused at build time. Fix: `wouldFormBridge` now enumerates `findAutoBridges(idx)` candidates and tests each with the new **pure** `bridgeWouldForm(row)` predicate, which mirrors `tryFormBridge`'s accept/reject (incl. collision + the red-can-cut-scoring asymmetry) **without mutating** `activeBridges`/SVG. Net: highlight ⟺ build, so a turquoise dot is guaranteed placeable. `findBridgeContaining` is now **unused** (kept for reference). Don't revert `wouldFormBridge` to it. Verified with a Node harness over the real extracted functions (round-count + S2 fallback + honest-negative + 5-column multi-bridge).
- **Middle-stone shadowing — `findAutoBridges` now enumerates EVERY position (Walter, 2026-06-02; the 2026-06-01 fix above was incomplete):** the longest-only fix made each *end* push both its 4- and 3-row, but the **middle** row (idx in the bridge interior) was still only a `!anyEnd` fallback that returned the **first/longest** middle row. So a buildable bridge with the placed stone in the **centre** got shadowed two ways: (a) any valid *end-anchored* candidate on that line set `anyEnd=true` → the middle fallback was skipped entirely; (b) even when it ran, a collision-blocked 4-row was returned before the buildable contained 3-row. Walter, 2026-06-02: *"It is a 3 bridge from the lower left purple to the upper right orange ... was not accepted"* — the **red stone he placed was the CENTRE** of purple-RED-orange (an anti-diagonal), and a blocked collinear end/4 candidate hid it, so the dot refused the tile. **Fix:** since auto-form is gone, `findAutoBridges` is purely the *"does any legal bridge run through this dot?"* enumerator for `wouldFormBridge` — it now collects **every** valid 3-/4-row containing `idx` at **any** offset (first, interior, last), in all 4 directions, and `bridgeWouldForm` filters. This is a strict **superset** of the old candidate set (it only *adds* rows, never removes), so previously-accepted dots can't regress and honest negatives can't false-highlight — highlight ⟺ buildable still holds. The `anyEnd`/middle-fallback branch is **deleted**. Reproduced + fixed + regression-checked (end-anchored still accepts, honest-negative still refused, plain & blocked middle now accept) with a Node harness over the real functions.
- **Refused manual bridge now says WHY (Walter, 2026-06-02 — "Keine Ahnung warum"):** a rejected double-click used to only red-flash the stone (`flashStoneInvalid`, the 2026-05-27 choice) with no explanation, so a *correct* refusal (e.g. red crossing a red/cut bridge, mixed-colour row, would-extend) looked like a mystery. `tryFormBridge` now sets a module-level `bridgeRefuseReason` on **each** of its 6 reject paths (not-straight / wrong-length / invalid-colours / shares-2+ / red-crosses-red-or-cut / scoring-crosses) and clears it on success; `handleStoneDblClick` shows it in a transient `#bridge-hint` banner under the turn indicator (`showBridgeHint`, auto-hides ~5 s) right beside the flash — and on the 3rd-click guard ("two already selected"). The flash stays; the reason is **additive**. Strings are localized (DE in `divided_loyalties.html`, EN in `_en`); comments are byte-identical across variants. This was Walter's chosen fix (via AskUserQuestion) — he declined relaxing the red-vs-cut rule, so **no rule changed**, only feedback.
- **Nobody crosses an existing bridge (Walter, 2026-05-27):** red bridges are pushed into `activeBridges` with `player:'red'` (previously they only existed as SVG lines), so collisions can see them. Both branches of `tryFormBridge` test crossings. **Red branch (Walter, 2026-06-02 — cuts ALL crossings):** `bridgesCrossedBy(c, row)` returns *every* bridge the red line crosses; if any is `cut` or `player==='red'` the whole red placement is illegal → `return` (nothing cut; "Rot kann Rot nicht schneiden", and **a red bridge can never be cut by anything**); otherwise `destroyBridge` **each** crossed scoring bridge. This replaced a single-`checkCollision` call that cut only the FIRST crossing — a red diagonal slashing a blue **+** (a horizontal *and* a vertical bridge) cut one and left the other on the board *and still scored* ("the cut bridge is still scored"). **Scoring branch:** any collision → `return` (Walter: "the yellow or blue bridges cannot cross over the red ones — nobody can"). The defensive value of red comes from cutting at **construction time** only; once a red bridge is on the board, scoring lines must route around it. `checkCollision(c, skipRed, newRow)` is now a thin "first crossing only" wrapper over `bridgesCrossedBy` (used by the scoring branch + the dot-highlight `bridgeWouldForm`); `skipRed` is always `false` from all call sites (kept in case a future rule needs the asymmetry). `recomputeScores` filters out **both** `b.cut` and `b.player === 'red'`, and the red branch ends in `updateScore()`, so a cut bridge stops scoring immediately.
- **Crossing/cover detection — GRID-EXACT, no floating point (Walter, 2026-06-02; "yellow and blue bridges cannot cut OR cover ANY other bridge, and red cannot cover other red"):** the float `ccw` segment test was scrapped — it kept missing through-stone crossings *and* (because of the 0.5-cell endpoint offset) some near-endpoint diagonal crossings, producing "yellow covers blue / yellow covers red" (both bridges drawn crossing, both still scoring). Replaced by an integer method:
  - `bridgeDoubledInfo(row)` maps a straight bridge to its **doubled-grid** occupancy: every **stone center** (doubled coords land on even,even) plus every **edge midpoint** between consecutive stones (carries an odd component). The two kinds can never collide, so a shared doubled point is unambiguous. Each point is tagged `{stone, end}` (`end`=one of the bridge's two endpoint stones).
  - `bridgesCrossedBy(newRow)` returns **every** active bridge the candidate crosses/covers. Conflict iff a shared doubled point is **(a)** an **edge midpoint** (→ a mid-gap diagonal crossing *or* a collinear cover) or **(b)** a **stone interior to BOTH** bridges (→ an X/+ through a shared stone). Legal (no conflict) only at a **junction**: a shared stone that is an **endpoint of at least one** bridge — fan-out, T, L, or end-to-end collinear (the 5-stone column). This exactly matches the **drawn** geometry (the 0.43-edge offset never reaches an edge midpoint), **verified by a brute-force harness: 0 mismatches across all 88,506 length-3/4 bridge pairs vs an independent exact segment-intersection oracle on the offset coordinates.**
  - `checkCollision(newRow, skipRed)` is now just "first conflict" over `bridgesCrossedBy` (scoring placement + dot highlight). **Red branch** (`tryFormBridge`): `bridgesCrossedBy(row)` → if any conflict is `cut` or `player==='red'` the whole placement is illegal (red can't cover red/cut, and **red is never cut by anything**); else `destroyBridge` **each** crossed scoring bridge. **Scoring branch:** any conflict → blocked. All call sites pass `row` (no offset coords needed for collision; `coords`/`c` is still computed only for `drawBridge`), so highlight ⟺ build.
  - **A perpendicular + through a shared *center* stone (interior to both) is BLOCKED for scoring** — this is the intended override of the older "perpendicular crosses are legal / single-stone overlap" notes. A stone may still anchor multiple bridges as long as they meet as **junctions** (endpoint-shared), not through-crosses. `sharesTwoPlus` (still upstream in `tryFormBridge`/`bridgeWouldForm`, skipping cut bridges) is now largely subsumed by the edge-midpoint rule but kept as an explicit "can't extend/re-form a row" guard.
- Cut bridges stay in `activeBridges` with `cut:true` — score deducted, but still block new bridges (`checkCollision` walks them).
- Score split: `bridgeScore` (3=1pt, 4=2pts) + `regionScore` (first scoring bridge per player×board whose *inner* cells touch a region = +1).
- Undo + save: `undoStack` snapshots before each user-input mutation; `localStorage["dl_savegame"]` auto-saves; `▶ Letztes Spiel fortsetzen` in setup; manual JSON export/import + WhatsApp share.
- Editor: custom positions with independent W×H, brush bar of board tokens + Punkt + eraser, persisted to `localStorage["dl_custom_positions"]`. CSV export + WhatsApp share use shared `buildPositionCSV()`.
- Stones nearly fill the cell (`--tile-size: calc(var(--cs) - 4px)`). Bridge endpoints offset by `0.50·cellSize` (don't graze stones). Bridges over stones (`#bridge-layer z-index:20`, `.tile z-index:15`).

**Don't touch:**
- **No `wa.me` redirect** in any share-fallback path (`shareCSVViaWhatsApp`, `shareGameJSONViaWhatsApp`, `shareOnWhatsApp`). Android WebView can't dispatch `whatsapp://` → `ERR_UNKNOWN_URL_SCHEME`. Fallback = download + alert.
- **No `text:` field** on `navigator.share` calls that pass `files:` — Android WhatsApp drops the file when both are present.
- **Don't "rebalance" `BLUE_BATCH` / `YELLOW_BATCH`** — Walter confirmed the per-batch asymmetry (yellow gets 1 purple vs blue's 3) is intentional role-symmetry.
- **Color brightness hierarchy:** blue/yellow may be bright; red/violet/orange stay darker (red = "evil" defensive color); green darkest. Bridges always one Material step darker than their stone.
- **Board grays:** `S→#3a3a3a` (darkest), `W→#6a6a6a` (middle), `G→#9a9a9a` (lightest) — not monotonic by cell count, per Walter's order.
- **Don't reintroduce auto-dots** in the editor — Walter places them explicitly with the Punkt brush.

### `makalaina.html` / `makalaina_remote.html` — MAKA LAINA
2-player disc collection on 12×12. 54 discs (36 primary numbered 1–6 in R/B/Y, 18 secondary in G/O/P). Draft system: after each collection, draw 1 from bag and place in your LOS. Sacrifice/relinquish mechanic when stuck (relinquish capped at 2/player). Winner = most colors won (tiebreak: highest single-color score). Remote uses PeerJS with full state sync.

- **Opening flow (Walter's rule — fair under any disc layout):** 24 random discs are dropped onto the empty board first. Then **Player 1 places both collectors** (one click → Black, second → White). Once both are down, **Player 2 picks which colour they want** — Player 1 plays the other and goes first. State machine: `SETUP → PLACEMENT → P2_CHOOSE → PLAY`. In the remote variant, host = P1 (the only one allowed to click during PLACEMENT); the joiner = P2 picks via `p2PicksColor()` which sends a `p2_pick_color` PeerJS message + a follow-up `state_sync`. **No curated starting positions** — Walter removed them; opening fairness comes from P2's colour pick, not from balanced disc layouts.
- **Resume last game — `▶ Resume last game` (Walter, 2026-06-30; local only, mirrors DL's „Letztes Spiel fortsetzen"):** a mis-tap by 2mm could lose a whole game, so the running game now auto-persists to `localStorage["ml_savegame"]` and a teal resume button + 🗑 discard appear at the top of the setup panel (`#ml-resume-slot`, rendered by `mlRenderResume()` at startup). `mlSaveGame()` serialises `{version, currentPosition, showLOS, moveHistory}` and is called from the tail of **`saveSnapshot()`** (every committed move) and **`undoMove()`** — so it captures exactly the same committed states undo does (`PLAY`, `DRAFT_PLACE`, `SACRIFICE_MOVE`, `END`; `SACRIFICE_SELECT` is never a tail). It stores the **full `moveHistory`**, so **Undo survives a resume too** (DL by contrast resets its undo stack). `mlResumeSavedGame()` restores `moveHistory` + live vars (via `mlRestoreLiveFromSnapshot()`, a field-for-field copy of `undoMove`'s restore block kept separate so undo's proven code stays untouched), hides setup/starter panels, shows `#game-actions` + `#csv-export-btn`, re-shows the draft panel iff resumed into `DRAFT_PLACE`, syncs the LOS toggle, then `updateUI()`+`render()`. The first `saveSnapshot` of a **new** game (the `p2PicksColor` baseline) overwrites the old save. **Remote has none** — `makalaina_remote.html` is live PeerJS multiplayer, no local-save model (same scope limit as undo).
- **Undo:** `↶ Undo move` button in `#game-actions` (local only — `makalaina_remote.html` has no undo). Every `saveSnapshot(desc)` captures full state (`cells`, `bag` contents, `gameState`, `scores`, `playerFlags`, `collectors`, `collectedDiscs`, `relinquished`/`relinquishCount`, `draftChosen`, `pendingExtraTurn`, `selectedIdx`, `sacrificeRange`, `sacrificingPlayer`, `isFirstMove`). `undoMove()` pops the last snapshot and restores live state from the new head, exits review mode. Disabled when `moveHistory.length <= 1` or `gameState === 'END'`. `applyPositionData()` and `startRandom()` both call `resetGameState()` so test-playing from the editor after a prior game starts fresh.
- **Snapshot timing (Walter bugs, 2026-05-29 — fixed in both variants):** `processCollection` now snapshots at the *resolved* terminal state of the turn, not mid-way. (a) **Draft branch:** snapshot is taken *after* `gameState='DRAFT_PLACE'` + `draftChosen=bag.pop()`, so undoing a draft *placement* lands back in DRAFT_PLACE with the disc in hand (`undoMove` re-shows the draft panel when the restored state is `DRAFT_PLACE`), ready to re-place — previously it dropped to PLAY with no disc. (b) **No-draft (bag-empty) branch + `relinquishTurn`/`performRelinquish`:** snapshot is taken *after* the `currentPlayer` switch, so the saved state records whose turn it now is. Snapshotting *before* the switch made undo restore the previous mover as current player — the "it's the other player's turn after undo" bug, which only surfaced once the bag was empty (with a draft, the post-placement snapshot masked it).
- **Free escape move (Walter's rule, 2026-05-29):** a trapped player (`hasValidMove`==false) with **no primary discs to sacrifice** no longer forfeits. `freeEscapeMove(player)` grants a free collector move of up to **2** squares (sets `SACRIFICE_MOVE` + `sacrificeRange=2`, no disc lost, no opponent pick), reusing the existing `SACRIFICE_MOVE`/`getReachableCells`/`handleSacrificeMove` plumbing. Falls back to relinquish/end only if even a 2-square move is impossible (truly boxed in). Applied at **both** trigger points in each file: the manual `sacrificeTurn`/`doSacrifice` no-discs branch *and* the auto-forced-end inside `updateUI`. Rationale (Walter): being trapped early used to be a near-certain loss; a guaranteed escape removes the incentive to trap the opponent in the opening. The "2" is a tunable constant — change it in `freeEscapeMove` in both variants.
- **Custom-position editor (local only):** `mlOpenEditor(null)` opens the 12×12 painter. Brush tokens — `R1..R6`/`B1..B6`/`Y1..Y6` for primary discs, `G`/`O`/`P` for secondaries; paint keyed by `y*12+x`. Save → `localStorage["ml_custom_positions"]` as `{customId, d:[[x,y,color,num],...]}`. **ML positions are nameless (Walter, 2026-05-17):** unlike DL — where the name describes a recognisable *shape* (Pyramid looks like a pyramid, "E M 3 W" spells the letters, Frankenstein resembles one) — ML layouts have no shape, so there is no name field, no name prompt, and no name in the saved JSON / CSV / share. Customs are shown numbered ("★ Position 1", "★ Position 2", …) in setup-order; the H1 shows just "MAKA LAINA" (no `— name` suffix) when a custom is played. Legacy customs that still carry a `name`/`b`/`w` field load fine — those fields are ignored. **Collector brushes (Walter, 2026-05-19):** the editor has two extra brushes — `CB` (Black = Player 1) and `CW` (White = Player 2), rendered as gold-bordered rounded squares like the board collectors. ### `capovolto.html` — Capovolto
Othello on a polyomino grid. Place 18 polyominoes, then alternate numbered discs across 6 rounds. Flip 4-disc lines (2 of each color); opponent's last value sets your direction (even=H/V, odd=diagonal). Scored by value majority per polyomino.

### `rainbow_blackjack.html` (+ `_en`, `_remote`) — Rainbow Blackjack
2-player tower-building to 21. 36 colored stones + 6 gray jokers. Each colored stone can go in its own tower or the two color-wheel neighbors. Jokers copy previous value; can't be first or consecutive. First to own 3 towers at exactly 21 wins instantly; else most towers. Remote uses PeerJS. Three Share-on-WhatsApp button instances (setup/game/scoring `<h1>`s).

### `frankenstein.html` — Frankenstein
Memory game, 1–4 players. Flip cards to build a complete body from torso outward (or reversed). 32/48/64 cards for 2/3/4 players. Strict torso adjacency per limb; head needs a top torso. Build modes: Classic (torso→out), Reverse (extremities→in), Mixed (alternate per player). Color-specific head designs (green/orange/purple/blue) with mix-and-match ear halves. Heads span 2 grid columns. "View Frankies" at game end before new game.

## Share on WhatsApp button

Every game has a Share-on-WhatsApp button right of the `<h1>` (green `#25D366`, `gap:14px`).

**Button LABEL is localized per file language (Walter, 2026-06-30):** DE → `Auf WhatsApp teilen`, EN → `Share on WhatsApp`, JP → `WhatsApp で共有`, CN → `分享到 WhatsApp`, UA → `Поділитися у WhatsApp` (brand "WhatsApp" kept in every language). Applies to all share buttons in a file (rainbow_blackjack has 3). When adding a localized variant, set the button text to match `<html lang>`. (The DL share *dialog* heading + rules reference use the same localized term.) **Caution:** do the non-ASCII replacements with a UTF-8-safe tool (Python/Edit), NOT `perl -CSD -pe 's/…/<cjk>/'` — the literal in the `-e` program gets double-encoded into mojibake.

`shareOnWhatsApp()` flow:
1. If `window.location.protocol === "file:"`, rewrite to `https://game.ywesee.com/parados/<filename>` (WebView container paths are useless to recipients). **Keep this guard for any new game.**
2. Try `navigator.share({ text: msg })` first.
3. If `navigator.share` is missing, copy `msg` to the clipboard and `alert()` the user to paste it into WhatsApp. **NEVER `wa.me` (Walter, 2026-05-31).**

**No `wa.me/?text=` anywhere — text shares too (Walter, 2026-05-31):** the old fallback redirected to `whatsapp://send/?text=…` and Android WebView (and some in-app browsers) can't dispatch the scheme → `net::ERR_UNKNOWN_URL_SCHEME` ("Web page not available"). The replacement is the same pattern as file shares: try `navigator.clipboard.writeText(msg)`, fall back to a hidden `<textarea>` + `document.execCommand('copy')`, then `alert()` a localized "Message copied — paste it into WhatsApp" (DE/EN/JP/CN/UA per file `lang=`). If even the textarea copy fails, `prompt(note, msg)` shows the text so the user can long-press to copy. Applied to all 17 game files (`democracy×3`, `kangaroo×5`, `divided_loyalties×2`, `makalaina×2`, `capovolto`, `frankenstein`, `rainbow_blackjack×3`). The `navigator.share` short-circuit stays first so iOS/Mac/desktop browsers keep the native share sheet — only the no-share fallback changed.

For file shares (CSV / saved game JSON), use `navigator.share({ files:[file], title:'…' })` — no `text:`, no `wa.me` fallback. See divided_loyalties guardrails.

## Hash deep-links per position

`divided_loyalties.html` (+ `_en`) and `kangaroo.html` (+ 4 language variants) accept a URL hash that auto-launches a specific starting position on load and on `hashchange`. The pattern: `findPositionByHash(hash)` resolves either a slug or a numeric index, and `setPositionTitle(label)` updates an inline `<span id="position-title">` in the H1 plus `document.title`. `shareOnWhatsApp` appends the active position's slug to `shareUrl` and includes the position name in the message body.

**Cross-language slugs:** kangaroo levels use English mnemonic slugs (`first-steps`, `the-block`, `shifting`, `staircase`, `scattered`, `slant`, `the-cross`) stored in `LEVEL_DATABASE[*].id` — identical across all 5 language variants, so `kangaroo_jp.html#the-cross` and `kangaroo_en.html#the-cross` resolve to the same level. Numeric `#1`-`#7` works as shorthand by `LEVEL_DATABASE` index. Trailing `b` selects start variant B (`#the-crossb`). The `findLevelByHash` regex `[a-z0-9-]+?` allows dashes inside the slug. **Don't switch slugs to localized names** — that would break cross-variant link sharing.

**makalaina has no hash deep-links** — curated positions were removed (Walter's fairness rule lives in the opening sequence, not in disc layouts). `currentPosition` is still passed in the PeerJS `state_sync` payload but no longer drives any title or URL state.

**Kangaroo branding per language — DUK is German-only (Walter, 2026-05-31):** `DUK` = **D**as **U**ngeduldige **K**änguru, an acronym that only parses in German, so it lives **only** in `kangaroo.html` (title, H1, `Lade DUK...`, `document.title`, share message). The other variants drop it:
- **EN** (`kangaroo_en.html`) → rebranded **`TIK`** (The Impatient Kangaroo) in H1, `<title>`, `Loading TIK...`, `document.title` prefix, and share message.
- **JP** (`kangaroo_jp.html`) → full name `せっかちなカンガルー`, no acronym.
- **CN** (`kangaroo_cn.html`) → full name `急躁的袋鼠`, no acronym.
- **UA** (`kangaroo_ua.html`) → full name `Нетерпляче Кенгуру`, no acronym (loading text is just `Завантаження...`).
- **`index.html`** overview card heading uses the English **`TIK — The Impatient Kangaroo`**.
**Don't reintroduce `DUK` outside `kangaroo.html`** — it confused non-German players. Cross-language URL **slugs** are unaffected (they're English mnemonics, see above); only the display branding is localized.

**Localized share messages:** kangaroo's `shareOnWhatsApp` message is localized per variant. DE uses "Spiele DUK — Das ungeduldige Känguru — Stufe X: …" + "Hol dir die Parados-App:"; EN uses "Play TIK — The Impatient Kangaroo — Level X: …" + "Get the Parados app:"; JP, CN, UA each have their own localized verb and full game name. `setPositionTitle`'s `document.title` prefix is localized too (e.g. `せっかちなカンガルー — …` in JP).

**Anchor links reference:** `docs/parados_anchor_links.pdf` lists every direct-entry URL across all games with clickable hyperlinks (60 anchors total: 12 DL-DE + 12 DL-EN + 7×5 kangaroo + repo link — DL has 12 starting positions as of 2026-05-16: a 4-round set (ids 1–6) + a 3-round set (ids 7–12)). **Regenerate via `docs/generate_anchor_links.py`** (Walter, 2026-05-31): it reads position names + level titles **live** from the game HTML (DL embedded CSV + kangaroo `LEVEL_DATABASE`), so renames/branding never drift — run `python3 docs/generate_anchor_links.py` then render the emitted HTML to PDF with **Chromium** (honors the CSS exactly + emits clickable link annotations): `chromium --headless --no-sandbox --no-pdf-header-footer --print-to-pdf=docs/parados_anchor_links.pdf "file://$(pwd)/docs/parados_anchor_links.html"`. The script prints a hyperlink count (must be 60). Layout: A4 landscape, every table shares one fixed `table-layout` grid (key 11% / label 31% / URL 58%) so columns line up vertically across all pages; header bands + zebra striping. URLs are `<a href>` tags (required for clickable `/URI` annotations). LibreOffice (`soffice --convert-to pdf:writer_pdf_Export`) still works as a fallback but renders as a loose Writer/Web doc and ignores most styling. Both the generator and the intermediate `parados_anchor_links.html` are checked in.

**`#rules` deep-link (Walter, 2026-05-28):** every game variant accepts `#rules` and opens its rules modal on load + `hashchange` — so the rules can be shared per game *and* per language by URL, e.g. `kangaroo_jp.html#rules` opens the Japanese DUK rules, `divided_loyalties_en.html#rules` opens the English DL rules. Two patterns matched the two existing rules APIs: games using `openRules()` (kangaroo×5, capovolto, frankenstein, rainbow_blackjack×3) get `if (location.hash === '#rules') openRules()`; games using `toggleRules(show)` (DL×2, makalaina×2) get `toggleRules(true)`; democracy×3 set `rules-modal.style.display='flex'` directly because their `toggleRules()` toggles without args. The handler runs once at script tail + on `hashchange`. For files with existing hash logic (DL, kangaroo, makalaina local), `#rules` is folded into the existing dispatcher so it doesn't race with position/level/editor hashes; for the rest it's a small block before `</script>`. `index.html` and `startpositionen.html` are intentionally excluded (no game rules of their own).

## Startpositionen hub (`startpositionen.html`)

Standalone, self-contained page (Parados dark theme) — **not a game**, a tool: one place to create/share starting positions. Scope is **DL + ML only** (the only games with shareable position CSVs; kangaroo levels are code-defined, the other 4 games have no position concept). It does **not** re-implement any editor — it deep-links into the existing ones and centralises the *send* step. Linked from `index.html` footer.

- **Disc-edit help:** the ML card has a collapsible native `<details class="help">` ("Wie verschiebe ich 1 Disc?") explaining that editing a disc/collector = changing `x,y` (0–11) in its CSV row, the 3 load rules, and the CSV-edit path. (Note: as of 2026-05-19 the ML editor *also* has collector brushes, so openings can be authored/edited in the editor too — the help text predates this and still describes the CSV path, which remains valid.) No JS — pure `<details>`.
- **Deep-links:** buttons open `divided_loyalties.html#editor` / `makalaina.html#editor`. Both games gained a tiny `startup()`/load hook: `if (location.hash === '#editor')` → `openEditor(null)` (DL) / `mlOpenEditor(null)` (ML). Keep these hooks if touching those load paths.
- **"CSV laden & erneut senden":** load/paste any exported CSV → `detectCSV()` sniffs the header (`position_id,position_name,row,c0…` = DL; `position_id,element,x,y,color,number` = ML, with a `collector_(black|white)` row → ML opening) to pick filename/title → 📤 share / 📋 copy / 💾 download. Same guardrails as the games: `navigator.share({files:[…]})` only (no `text:`), no `wa.me`; fallback = download **+** clipboard copy + alert.

## Server-side

`cgi-bin/save_csv.py` (Python 3 CGI): accepts POST `{game, csv}`, saves to `csv/` as `gamename_HHMM_dd.mm.yyyy.csv` (collision-avoidance suffix). `csv/` owned by `www-data`. Apache: `cgid` module with `ExecCGI` on `cgi-bin/`.

Game CSV exports (`downloadCSV()`) POST to this endpoint and also trigger a local browser download.
